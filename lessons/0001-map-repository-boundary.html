LESSON 0001 · 约 30 分钟

建立 lockstep_bench 仓库地图

本课的胜利条件是能准确说明：哪些内容属于正式仓库，哪些是构建产物，哪些是外部原型依赖，以及发生设计冲突时应信任什么。

先固定绝对边界：学习根目录是 D:\0_ongoing\lockstep\lockstep_bench。同级 D:\0_ongoing\lockstep\software 不属于课程。

1. 认识正式仓库的两大部分

lockstep_bench/
├─ docs/                         项目级设计与模块设计
└─ software/lockstep_host/       正式上位机工程
   ├─ src/                       C++/Qt 产品源码
   ├─ resources/ 与 profiles/    固化运行资源
   ├─ tests/                     自动化规则验证
   ├─ scripts/ 与 packaging/     构建、审计和交付
   └─ docs/                      实现层补充说明

仓库内也有一个名为 software 的目录，但它的完整路径包含 lockstep_bench。判断边界时必须看完整路径，不能只看末级目录名。

2. 区分五类信息

类别位置阅读目的
项目设计docs/开发文档/理解职责、约束、状态与验收标准
正式实现software/lockstep_host/src/确认设计当前如何落地
运行合同resources/、profiles/理解板卡、trace、模板和 manifest
可执行规则tests/确认哪些不变量被自动验证
交付机制scripts/、packaging/理解如何构建、审计和部署产品

3. 识别不属于设计本体的内容

  • build-* 是本地构建目录，可以帮助运行和诊断，但不是源码真相。
  • .codex-tasks/ 是任务过程与实验产物，不代表正式架构。
  • .Xil/、Vivado 日志和备份日志是工具产物。
  • 工作区未提交修改可能比文档更新；理解当前行为时必须同时核对源码、测试和 Git 状态。
  • 同级 D:\0_ongoing\lockstep\software 是功能原型机，不用于解释正式实现。

4. 找出一个重要边界风险

正式仓库的 software/lockstep_host/src/apps/CMakeLists.txt 当前仍设置：

set(LOCKSTEP_FAULT_SCRIPT_DIR
    "${CMAKE_SOURCE_DIR}/../../../software/error_inj")

这会在构建后从仓库外的原型目录复制注错脚本。学习时不进入原型目录，但评审时应能指出：正式工程并未完全自包含，换机器或独立检出仓库后可能缺少构建输入。这是“边界泄漏”，不是课程资料来源。

5. 建立权威性判断顺序

当资料不一致时，不要简单地认为“文档永远正确”或“代码永远正确”。按问题类型取证：

  1. 问设计意图：读已裁决的模块设计说明和待裁决问题汇总。
  2. 问当前行为：读当前源码和运行资源。
  3. 问规则是否受保护：读自动化测试并实际运行。
  4. 问能否独立交付：检查 CMake、脚本、资源清单和仓库外引用。
  5. 发现冲突：明确记录“设计意图、当前实现、测试覆盖”三者差异。

6. 动手练习

  1. 只在 lockstep_bench 内，为上述五类信息各找一个具体文件。
  2. 说明 lockstep_bench/software/lockstep_host 与 lockstep/software 为什么不是同一个目录。
  3. 从 src/README.md 选出三个业务模块，用一句话说明各自职责。
  4. 用自己的话解释仓库外注错脚本引用为什么是维护风险。

完成后，把“五类文件各一个 + 三个模块职责 + 外部引用风险”发给我。第二课将进入正式程序入口，建立第一个 UI 到任务落盘的纵向调用链。
