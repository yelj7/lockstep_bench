程序如何启动并响应操作

本课结束时，你应能沿正式源码讲清：可执行程序从哪里进入、为什么不会立即退出，以及按钮操作如何到达业务控制器。

只记一条主线：main() 创建对象 → exec() 等待事件 → 窗口 emit 动作 → 控制器 connect 接收 → handleAction() 分派。

1. 一个 EXE，三种入口模式

正式入口位于 software/lockstep_host/src/apps/ui_preview_main.cpp:252。main() 先检查命令行参数：

参数入口函数用途
--offline-capturerunOfflineCapture离线采集处理
--live-capturerunLiveCapture实时采集处理
--local-server 等runDebugServiceMode调试服务模式
没有上述参数继续执行 main图形界面模式

设计含义：产品虽然只有一个可执行文件，但入口层承担多个运行角色。维护时遇到问题，第一问应是“当前走的是哪种入口模式”。

2. GUI 模式创建四个关键对象

QApplication application(argc, argv);
lockstep::ui::MainWindowShell window;
lockstep::ui::SplashWidget splash;
lockstep::apps::WorkbenchController controller(&window, mode, &application);

把每行按“类型、变量、构造参数”阅读：

  • QApplication 管理 Qt 应用和事件循环。
  • MainWindowShell 是主窗口，负责展示和发布用户动作。
  • SplashWidget 是启动页。
  • WorkbenchController 是协调器；&window 把窗口地址交给它，&application 作为 Qt 父对象。

这里的 Type variable; 是创建对象，&variable 是取得对象地址。暂时无需深入栈、指针运算或内存布局。

3. 为什么窗口会持续响应

return application.exec();

exec() 启动 Qt 事件循环。程序随后持续等待并分发按钮、定时器、窗口、串口等事件，直到退出。Qt GUI 不是按源码从上到下反复执行，而是“初始化一次，然后由事件驱动”。

4. 窗口把操作包装为数据

ui_common/ui_contract.h:22 使用 enum class UiAction 列出所有动作，例如 NewTask、ProgramImage 和 AnalyzeProtocol。它们是受类型约束的动作编号。

MainWindowShell::emitAction() 在 main_window_shell.cpp:3556 创建请求：

UiActionRequest request;
request.action = action;
request.page = page;
request.objectName = objectName;
emit actionRequested(request);

窗口没有直接创建任务或操作硬件。它只发布“发生了什么”以及必要参数。这份 UiActionRequest 就是 UI 与控制器之间的数据合同。

5. 控制器订阅信号

workbench_controller.cpp:837 建立连接：

connect(window_, &ui::MainWindowShell::actionRequested,
        this, [this](const ui::UiActionRequest& request) {
            handleAction(request);
        });

逐段翻译：

  1. window_：信号发送者。
  2. actionRequested：要订阅的信号。
  3. this：当前控制器对象，也是处理上下文。
  4. [this](...){...}：信号到达时执行的匿名函数。
  5. handleAction(request)：把请求交给控制器分派。

6. 分派不是执行全部业务

handleAction() 位于同一文件 905 行。它先做忙状态阻断，再使用 switch 选择方法：

switch (request.action) {
case ui::UiAction::NewTask:
    createTask();
    break;
case ui::UiAction::SaveTask:
    saveCurrentTask();
    break;
// ...
}

因此完整链路是：

按钮点击→UiActionRequest→actionRequested
connect→handleAction→具体方法

这是后续所有 UI 业务追踪的统一入口。看到某个按钮时，先找对应 UiAction，再到 handleAction 找具体方法。

7. 维护与评审视角

  • 优点：UI 不直接依赖工作区、硬件和报告实现，动作合同集中。
  • 代价：WorkbenchController 集中了大量协调职责，新增功能容易让它继续膨胀。
  • 评审检查：新增按钮是否定义动作、携带必要参数、设置忙状态阻断，并被测试覆盖。
