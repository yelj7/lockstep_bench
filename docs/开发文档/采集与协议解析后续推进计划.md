# 采集与协议解析后续推进计划

## 1. 范围与原则

本计划只推进硬件采集、传输协议、上位机采集会话、协议解析、benchmark 和采样配置，不修改锁步核。CAN 深度语义、ETH、USB 等非当前必需能力只记录缺口，不在本轮扩展。

程序控制保留启动、终止、复位和运行，不增加暂停/继续类 stop 功能。内部 STOP_CAPTURE 仍作为采集终止和异常恢复命令，不等同于程序暂停。

所有 host 构建和测试写入 D:\tmp\lockstep。RTL 只在 78 服务器使用 Vivado 2022.2 XSIM；在仿真门禁充分通过且必须上板时才生成 bitstream。

## 2. 首轮审计结论

## 3. 推进阶段

### 阶段 A：完善 benchmark 和门禁

- [x] 增加 host 结束原因失败矩阵，禁止 watchdog、hard timeout 和 fatal 误报成功。
- [x] 增加 32-bit sample index 回绕和事件早于窗口的时间轴门禁。
- [x] 修正 VCD JTAG Shift 边界并更新经过审查的 golden。
- [x] 增加 RTL 非零绝对索引、跨回绕和连续两次 ARM 的时间轴测试。
- [x] 增加 UART 正常帧、framing error、break、恢复和截断帧测试。
- [x] 生成并断言 SPI mode 0/1/2/3 的真实 CPOL/CPHA 波形。
- [x] 扩展 I2C ACK/NACK、repeated START、地址和方向组合断言。
- [x] 增加 JTAG TAP reset、1-bit、IR/DR 最后一位、Pause/Exit2 和截断边界测试。
- [x] 统一 JTAG golden 为 24 次 IR、24 次 DR，并修正位序和门禁矛盾。
- [x] 增加采样配置逐字段和不支持值拒绝矩阵。
- [x] 增加 ARM 到 DRAINING、EVENT_END、CAPTURE_END、CONFIGURED 的整顶层测试。

完成门禁：host 全量 CTest 和远端 XSIM 全绿，失败用例能够证明旧实现会误判或漏检。

### 阶段 B：完善采集与结束闭环

- [x] host 在线和离线回放共用结束判定。
- [x] 程序运行和程序复位并运行统一按 CONFIG、ARM、START_EVENT_STREAM、启动目标程序的顺序执行，再等待触发并执行可选注错。
- [x] 运行和独立下发均保存完整采集/事件线协议配置；打印 UART 不可用时不阻断启动，但缺失结束证据时禁止形成 pass。
- [x] 调试服务缺失、损坏或错配响应不再判成功。
- [x] 目标预检改为 RAM 原值写回校验，不再覆盖首条指令。
- [x] 已评估现有 program marker/MMIO/探针，未找到可靠的真实 `program_done` 来源；按本轮允许边界保持生产顶层绑 0，并禁止正常通过结论。
- [x] 实现 CAPTURING_PRETRIGGER -> CAPTURING_POSTTRIGGER -> DRAINING -> CONFIGURED。
- [x] program_done 后执行 quiet guard，停止接收新事件，再等待九路协议 FIFO 和跨时钟 FIFO 明确排空。
- [x] 保留首要结束原因，同时独立上报 overflow/dropped，避免原因覆盖。
- [x] 接入采样域掉线 fatal 路径，并把 last_error/device flags 传到 host。
- [x] 给 RX 命令解析增加截断帧超时和 magic 重同步。
- [x] 统一跨时钟 FIFO 两域复位合同并增加独立复位门禁。
- [x] 对当前硬件不支持的 rate/count/invert/mode/edge/event limit 等字段明确拒绝配置。

完成门禁：每种结束原因都有端到端帧证据；只有 program_done 且无 overflow/dropped/fatal 才能形成 pass。

- [x] UART：补 TX/RX、framing error、break、异常恢复和 EOF 截断；无格式元数据时明确 limited。

- [x] JTAG：统一 VCD/sparse TAP 边界，覆盖 1-bit、Pause/Exit2 延续和截断，不扩展指令语义。

## 4. 只记录不推进

- `can_txo/can_rxi` 的 AE12/AF12 位于 DDR4 使用、VCCO=1.2 V 的 Bank 65；`LVCMOS12` 只用于消除原 `LVCMOS18` 的 BIVC-1 冲突，不代表存在已验证的 CAN 收发器、连接器、布线或电平转换。外部 CAN 状态保持 `unsupported/design_gap`，禁止连接或驱动外部 CAN 设备，板测不验收 CAN RX/TX；只保留仿真和内部线级证据。
- CFG_GRETH=0，ETH 保持 unavailable/design_gap。
- 无 PL 可见 USB/ULPI 源，USB 保持 unavailable/design_gap。
- 真实 program_done 尚无可信 MMIO/探针来源，生产顶层继续绑 0；仿真只在 testbench 内强制该输入。
- SPI sparse 事件当前只记录 SCLK 上升沿，不能据此可靠区分四种 mode；补双边沿属于采集合同扩展，本轮不推进。
- JTAG IDCODE/SAMPLE/PRELOAD/EXTEST 指令解释与边界寄存器期望比较属于语义扩展，本轮只保证 TAP/IR/DR 现有解析正确。
- 锁步比较器和锁步核内部功能不在本计划范围。

## 5. 上板结果与后续边界

阶段 A 至 D 的可仿真门禁、远端 XSIM、debug/release 实现、timing、route、DRC、采集 CDC、bus-skew、I/O 和 SHA-256 已全部通过。c477 对同一版采集 RTL 完成了 FT601、CMSIS-DAP、benchmark、连续窗口、协议事件和统一时间轴的完整实板验证；e416 最终候选又完成了复烧、BSCAN/双 ILA 枚举以及 FT601 STATUS、CONFIG/ARM/STOP 冒烟。原始证据和边界见《[采集上板验证记录](./采集上板验证记录.md)》。

后续板测统一使用 `D:\tmp\lockstep\bitstreams-e4167da0\debug\noelvmp_zcu102_1024_debug.bit` 及同目录配对 LTX；release bit 只在不需要 ILA 时使用。不得连接 CAN 扩展板或任何外部 CAN 设备。真实 `program_done` 未接入，因此 hard timeout 必须继续判失败。CP2108 Interface 2 确实存在且是程序打印接口，但当前 Windows 将其绑定为 WinUSB 而非串口；修复驱动绑定前不得用 COM12/13/15 无输出替代 Interface 2 验收。
