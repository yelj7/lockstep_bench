# 触发窗口变化事件采集重构设计

## 1. 结论

连续窗口从 1024 bit 缩为 512 bit，只保留 `lockstep_ahb_sample_pack` 的完整 AHB 截面和 mismatch。UART、SPI、CAN、I2C、RV-JTAG 不再逐拍写入连续窗口，改由 18-bit 真实状态、4096 深共享变化环形缓存和 128-bit 紧凑记录覆盖同一触发窗口。18-bit 包含原 16 根同步线及真实 SPI CPOL/CPHA 配置状态。ETH、USB 没有可用真实探针，继续报告 `design_gap`，禁止生成事件。

AHB 与 mismatch 只以连续窗口作为权威来源，不再重复生成 sparse 事件。低速协议只以变化事件重建后的 VCD 波形作为协议解析来源。这样同时消除持续 AHB sparse 溢出和 Host 双来源重复计数。

## 2. 当前 1024-bit 映射审计

顶层当前拼接为：

```text
sample[511:0]    = lockstep_ahb_sample
sample[1023:512] = lockstep_protocol_probe
```

| 原位段 | 当前内容 | 删除依据 | 替代来源 |
|---|---|---|---|
| 519:512 | UART TX/RX/CTS/RTS、下降沿和 toggle | 低速重复采样；派生边沿可由状态变化得到 | 4-bit UART 状态变化 |

| 671:640 | ETH 状态 | 当前 `eth_core_enabled_i='0'`，不是可用真实探针 | `design_gap` |
| 735:704 | USB 常零 | 无真实探针 | `design_gap` |
| 767:736 | 普通 JTAG 与 RV-JTAG 两组线、rise/toggle/activity | 普通 JTAG 无板级 pad，是无效探针；只保留真实 PMOD1 RV-JTAG | 4-bit RV-JTAG 全变化 |

删除高 512 bit 不影响地址触发、mismatch 触发、AHB 事务解析或高速关联分析。

### 2.3 真实探针与时钟域核对

| 来源 | 顶层连接 | 结论 |
|---|---|---|
| AHB/mismatch | `noelvcore` 的 AHB 截面和 SafeLS comparator 直接进入 pack，均为 `clkm` | 真实、同 sample clock，无 CDC |
| UART | `uart_*_int` 经 switch(3) 在 DSU/UART 间复用并接板级 UART pad | 真实；也是 MI_02 程序打印物理路径 |
| SPI | core SPI 线经 J3 双向 pad | 真实 |
| CAN | `can_rxi_int/can_txo_int` 接板级 pad，当前 `CFG_CAN=1` | 真实内部 TX/RX；不扩展外部 CAN |
| I2C | SCL/SDA 使用 `iopad` 回读值 | 真实总线电平 |
| RV-JTAG | TCK/TMS/TDI/TDO 接 PMOD1 GPIO | 真实 |
| 普通 JTAG | core 内部信号没有顶层端口或 pad，输入悬空 | 无效，必须删除，不能伪装为第二组 JTAG |
| ETH | probe 输入虽接 RGMII 结构，但 `eth_core_enabled_i='0'` 且 `CFG_GRETH=0` | unavailable/design_gap |
| USB | probe 槽明确常零且没有输入端口 | unavailable/design_gap |

当前 `lockstep_protocol_probe_pack` 对异步线做了逐线两级同步，但缺少 `ASYNC_REG` 属性；这是必须修复的 CDC 实现缺口。JTAG 的 TMS/TDI/TDO 必须满足相对于 TCK 的协议稳定时间，Testbench 要覆盖双边沿与同拍变化；同步后的观测时刻作为权威时间戳，不对外部模拟边沿做不可证明的回推。

## 3. 低速同步状态映射

所有线先在 120 MHz sample clock 域经过带 `ASYNC_REG` 属性的两级同步，再参与变化检测。`state_after` 表示第二级同步器在该有效 sample index 观察到的值。

| state 位段 | 来源 |
|---|---|
| 0 | UART TX |
| 1 | UART RX |
| 2 | UART CTS_N |
| 3 | UART RTS_N |
| 4 | SPI SCLK |
| 5 | SPI MOSI |
| 6 | SPI MISO |
| 7 | SPI CS_N |
| 8 | CAN RX |
| 9 | CAN TX |
| 10 | I2C SCL pad readback |
| 11 | I2C SDA pad readback |
| 12 | RV-JTAG TCK |
| 13 | RV-JTAG TMS |
| 14 | RV-JTAG TDI |
| 15 | RV-JTAG TDO |
| 16 | SPI CPOL 配置状态 |
| 17 | SPI CPHA 配置状态 |

同步延迟不做主机侧猜测补偿：事件时间戳定义为“同步后的状态在 sample 域可可靠观察的绝对 sample index”。因此连续与 sparse 使用同一 sample 时钟和同一索引，没有额外协议相关偏移。窄于一个 sample 周期且未被同步器观察到的异步脉冲不在可恢复保证内。

## 4. 统一窗口合同

```text
trigger_index = T
window_start  = T - 2047
window_end    = T + 2048
有效区间      = [window_start, window_end]，共 4096 个 sample index
```

ARM 后必须先积累 2047 个有效预触发样本才允许接受触发。连续窗口和变化事件在同一拍接受触发，并在包含 `window_end` 的样本写入后同时冻结。STOP、触发 watchdog、内部错误和 overflow 保持不同失败原因；固定触发窗口正常完成不依赖 `program_done`。

窗口内样本槽位使用局部索引 `0..4095`，但 VCD 时间戳使用 `window_origin_tick + local_sample_index` 的 64-bit 绝对 sample tick，并按统一 timebase 转换。sidecar 保存 `window_origin_tick`、`window_start_tick`、`trigger_tick`、`window_end_tick_exclusive`、VCD timescale 和 counter generation。所有 64-bit JSON tick 使用十进制字符串，区间使用 `[start, end_exclusive)`；禁止把局部槽位索引当成 VCD 全局时间。

## 5. 128-bit 变化记录

共享 18-bit 状态使同一 sample index 的多协议变化合并为一条记录，不存在多源写端口互相覆盖。

| 位段 | 字段 | 说明 |
|---|---|---|
| 31:0 | absolute_sample_index | 32-bit 硬件 sample index，允许自然回绕 |
| 63:32 | change_sequence | ARM 后每次变化递增；窗口内必须连续 |
| 81:64 | state_after | 变化后 18-bit 真实状态 |
| 83:82 | reserved | 必须为 0 |
| 91:84 | source_mask | UART/SPI/CAN/I2C/ETH/USB/JTAG；ETH/USB 必须为 0 |
| 95:92 | reserved | 必须为 0 |
| 113:96 | change_mask | `state_before XOR state_after` |
| 127:114 | reserved | 必须为 0 |

四个 32-bit word 依次为 `absolute_sample_index`、`change_sequence`、`state_after + source_mask + reserved`、`change_mask + reserved`，任一字段不跨 word。`state_before = state_after XOR change_mask`，因此即使环形缓存覆盖了 ARM 初期旧记录，也能从最老保留记录恢复其前一状态。EVENT_META 发送窗口起始前一状态；Host 只应用窗口内按 `(absolute index, sequence)` 排序的变化。

EVENT_DATA payload 从 64 B 缩为 16 B；外层帧头和 CRC 保持不变。该不兼容记录语义使用 protocol v4，v2/v3 只用于读取旧证据，不允许新旧字段静默混用。

## 6. 容量与丢失证明

共享记录每个有效 sample 最多产生一条；4096-sample 窗口最多产生 4096 条。4096 深环形缓存因此覆盖最坏的“每拍任意多协议同时变化”。触发前允许覆盖窗口之外的旧记录；触发后最多 2048 条新记录，只能覆盖窗口之外的旧记录，不能覆盖 `[window_start, window_end]` 内记录。

冻结后才从 FT601 域读取，读口遵守 ready/valid 并允许任意背压，不需要活动采集期跨时钟 FIFO。若内部计数证明、帧序号、CRC、source mask、emitted/accepted/dropped 或容量不一致，必须置 fatal/overflow，并拒绝产品成功结论和开发门禁通过。

## 7. 当前实现资源基线

来源：source digest `e4167da022f30e7313ad1b8877b87d5b81daaa68961b39e34f06b18948a46d9c` 的 release utilization 报告。

| 实例 | LUT | FF | RAMB36 | RAMB18 |
|---|---:|---:|---:|---:|
| 完整 FT601 采集子系统 | 9619 | 7892 | 423 | 11 |
| 1024-bit 连续窗口 | 1282 | 838 | 112 | 6 |
| 九路 512-bit event FIFO | 3988 | 2181 | 254 | 5 |
| 512-bit event async FIFO | 171 | 151 | 57 | 0 |
| event encoder/controller/frame/timestamp | 904 | 503 | 0 | 0 |

理论存储位数：

```text
旧连续窗口 = 4096 * 1024 = 4,194,304 bit
新连续窗口 = 4096 *  512 = 2,097,152 bit
新变化缓存 = 4096 *  128 =   524,288 bit
```

仅按存储位数，新结构比旧连续窗口减少 1,572,864 bit；同时删除当前占 `311 RAMB36 + 5 RAMB18` 的旧事件 FIFO 链。预计采集子系统 RAMB36 从 423 大幅降至约 70 到 80，最终数值必须以改后 route utilization 为准，不以估算代替验收。

## 8. Host 单一来源规则

- 硬件 physical continuous 宽度为 512 bit；`capture.vcd` 保持逻辑 1024 通道以复用成熟的 AHB/协议解析通道位置。CH0..511 来自 continuous，低速线恢复到既有高位 raw-line 位置，其余高位明确标记 unavailable/constant，不伪装为 physical continuous。
- SPI CPOL/CPHA 分别恢复到既有 CH550/CH551，保证 analyzer 只从 merged VCD 取得逐事务 mode；sidecar 仅保留审计副本。
- 新 profile 使用 `trace.noelv.ahb512_sparse18_v4`，schema 分别声明 `continuous_sample_width=512`、`logical_vcd_width=1024`、`reconstructed_state_width=18` 和每段的 `source`。
- `capture_sidecar.json` 保存绝对时间原点、初始状态、原始变化记录、source mask、序号和统计。
- `protocol_analysis.json` 只从合并后的 VCD 解析协议事务；raw sparse 记录只作为 observation/audit，不再与 VCD 事务二次合并或替换。
- AHB/mismatch 的 canonical 来源固定为 continuous；UART/SPI/CAN/I2C/JTAG 的 canonical 来源固定为 reconstructed sparse waveform。
- 系统总体 implemented mask 为 `0x19f`；EVENT_META/END 的 sparse implemented mask 为 `0x09e`，AHB/mismatch 不应在事件统计中出现。
- sidecar、archive、逐事件与 ARM 返回的 capture ID 必须一致；任一冲突输出 `TRACE_SOURCE_CONFLICT` 或采集失败。

## 9. 必须验证的实现门禁

实现必须覆盖窗口三边界、回绕、连续 ARM、无变化初态、触发前/同拍/触发后变化、UART 残帧、SPI 四模式、I2C ACK/NACK 与 repeated START、JTAG 双边沿、多协议同拍、每拍变化、背压和故障注入。Testbench 原始同步状态与 Host 重建 VCD 必须逐 sample 完全一致。

在 Host、远端 Vivado 2022.2 XSIM、CDC、timing、route、DRC 和资源对比全部通过前，不生成 bitstream。
