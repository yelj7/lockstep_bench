# 1. 当前结论

目标连接、程序烧录、程序运行三个界面的 UI 方向已经基本确定；对应底层设计由 M05、M06、M07 承担，设计边界已经具备继续开发条件。

当前源码已经具备可用于 UI 联调的底层骨架：`DebugAccess` 抽象接口、`InMemoryDebugAccess` 内存后端、`TargetConnectionService`、`ProgramController`。本轮已补齐本机可完成的资源加载、阶段进度、证据落盘、ELF 可加载段解析、中止封装和任务证据恢复；这些实现能够验证连接 gate、预检 gate、烧写/回读/运行/中止的基本状态转换，但尚不等同于真实板卡可用版本。

# 2. 已完成范围

UI 已确定范围：

- 目标连接页不提供本机 `debug tool`、`interface.cfg`、`target.cfg` 路径输入。
- 目标连接页删除“高级连接设置”“加载目标配置”“保存目标配置”。
- 程序烧录页只关注程序烧写与回读，不包含程序运行、程序中止、运行摘要。
- 程序烧录页使用“烧写进度”和“回读进度”两个阶段进度条。
- 程序运行页复用烧录页的布局比例，只读显示已确定的镜像名称和路径。
- 程序运行页只提供“程序运行”“程序中止”，只显示运行摘要。
- 程序运行页使用“运行进度”和“中止进度”两个阶段进度条。

底层已具备范围：

- M05 已设计目标连接、断开、预检、连接 gate 和预检 gate。
- M06 已设计统一调试访问接口：连接、断开、状态、读、写、复位、运行、中止。
- M07 已设计镜像检测、写入、回读校验、运行 gate、运行控制和中止控制。
- 当前代码可识别或处理 ELF、SREC、BIN、Intel HEX；ELF 已按可加载段形成写入段。
- 当前代码可通过内存后端完成烧写、回读一致性比较、运行控制和中止控制记录。
- `resources/manifest.json`、profile、debug adapter 占位资源已能复制到 exe 同级并由资源管理器校验。
- profile 内调试工具和 cfg 路径已限制为资源包相对路径，不再接受本机绝对路径。
- 程序操作证据已使用固定文件名：`program_write_record.json`、`readback_verify_record.json`、`run_control_record.json`、`halt_control_record.json`、`program_operation_progress.json`。

# 4. 三个界面的局部开发计划

## 4.1 目标连接页

UI 开发：

1. 固化当前布局和交互：不显示本机路径，不显示高级配置，不允许滚轮改变下拉选项。
2. profile 下拉只显示逻辑名称、板卡型号、能力摘要和资源摘要。
3. 连接、断开、预检按钮根据当前状态启用或禁用。
4. 连接失败、预检失败以页面摘要和底部 Log 同步展示。

底层开发：

1. 已接入 M03 profile/resource loader，形成 `DebugProfile`。
2. 已调用 M05 `connectTarget`、`disconnectTarget`、`runPrecheck`。
3. 后续真实连接记录、预检记录需在真实后端返回后写入任务证据目录。
4. 已在连接和预检失败时关闭 M07 的烧写、回读、运行 gate。

验收标准：

- UI 中不存在本机配置路径输入框。
- 连接成功必须有真实后端返回或状态快照。
- 预检失败时程序烧录和程序运行入口不可执行。

## 4.2 程序烧录页

UI 开发：

1. 保持烧录页只包含程序镜像、烧录与回读、烧录与回读摘要。
2. 不提供格式、BIN 地址、Offset、运行地址覆盖。
3. 使用“烧写进度”“回读进度”两个进度条。
4. 进度条按阶段推进：可用性检查、镜像/范围解析、命令执行、结果确认、记录保存。

底层开发：

1. 已完善 M07 镜像识别和写入计划，ELF 已按可加载段解析。
2. 已接入 M06 `write`，按段烧写并保存返回；当前返回来自内存后端。
3. 已接入 M06 `read`，回读相同范围并比较；当前返回来自内存后端。
4. 已生成并持久化 `program_write_record.json`、`readback_verify_record.json` 和阶段进度记录。

验收标准：

- 不支持格式或地址越界时阻断烧写。
- 烧写失败时不得进入回读通过状态。
- 回读失败或不一致时程序运行 gate 关闭。

## 4.3 程序运行页

UI 开发：

1. 程序镜像区域只读显示烧录流程确定的镜像名称和路径。
2. 程序运行控制区只包含“程序运行”“程序中止”。
3. 摘要区只显示运行摘要，不提供回读/运行摘要切换。
4. 使用“运行进度”“中止进度”两个进度条。

底层开发：

1. 已在运行前检查 M07 gate：连接通过、预检通过、烧写成功、回读通过。
2. 已调用 M06 `run`，保存返回或状态快照；当前返回来自内存后端。
3. 已通过 M07 `haltTarget` 调用 M06 `halt`，保存返回或状态快照；当前返回来自内存后端。
4. 已明确运行确认只证明“调试控制命令被接受并有状态证据”，不证明程序业务功能正确。

验收标准：

- 回读未通过时运行按钮不可用。
- 运行成功必须保存真实返回或状态快照。
- 中止成功必须保存真实返回或状态快照。
- 若没有结果寄存器、串口输出、采集文件或协议解析证据，不得把程序业务功能正确性写成已证明。

# 5. 推荐实施顺序

1. 已完成 M03 profile/resource loader 与目标连接页绑定。
2. 下一步必须实现 M06 真实调试后端的最小闭环：连接、状态、读、写、复位、运行、中止。
3. 将 M05 目标连接和预检切到真实后端，并完成一次上板连接验证。
4. 使用真实板卡验证 M07 的 ELF/SREC/BIN/Intel HEX 写入计划和阶段进度。
5. 使用真实板卡验证程序烧录页与 M07 的真实写入、回读、记录落盘绑定。
6. 使用真实板卡验证程序运行页与 M07/M06 的运行、中止、状态快照绑定。
7. 最后接入 M13 测试报告读取烧写、回读、运行证据，形成测试模式 pass 结论。

# 6. 实现前需要准备的输入

- 真实 debug adapter 资源或自研调试服务接口，用于替换当前占位资源。
- 真实 profile 样例以及板卡可写 RAM 范围、默认运行地址、复位策略。
- 真实调试后端的调用方式：自研服务 API、动态库、命令行工具或临时 OpenOCD 适配。
- 真实板卡一块，用于验证连接、读写、复位、运行、中止。
- 至少一个 ELF、一个 SREC 或 S19、一个 BIN 样例镜像。

# 7. 当前局部结论

这三个界面可以继续并行开发 UI 与底层，但底层必须按“内存后端联调”和“真实上板验证”分两级验收。当前已经达到本机 UI 联调级，并补齐资源加载、阶段进度、证据落盘、ELF 段解析、中止封装和任务证据恢复。继续推进真实可用级时，已经进入必须依赖 M06 真实调试后端和上板验证的阶段。
