## 3. 未通过与边界

- Live capture 以 `EVENT_END reason=4 (hard_timeout)` 结束，Host 返回 25 并正确拒绝 pass；STOP 恢复成功。原因是生产顶层 `program_done` 固定为 0，程序虽已在 UART 事件中输出 `PROGRAM_RUN_DONE`，该标志尚未转换为硬件结束输入。
- CP2108 Interface 2 确实存在并作为程序打印接口；Windows PnP 将 Interface 0/1/3 以 `silabser/oem71.inf` 枚举为 COM13/COM12/COM15，却将 Interface 2 以 `libwdi/oem75.inf`、服务 `WinUSB`、类 `USBDevice` 绑定，因此 Qt `QSerialPortInfo` 无法枚举真正的打印接口。此前三个 COM 端口 0 bytes 不能证明程序未打印；完整接口证据位于 `D:\tmp\lockstep\board-validation\cp2108-interface-map-20260722.json`。
- 当前 Host 在 Interface 2 不可枚举时允许继续执行采集和程序启动，但记录 UART 结束证据缺失并禁止据此判 pass；待 MI_02 恢复绑定 `silabser` 后，Interface 2 会由现有优先级规则自动成为首选打印串口。
- 未设置 BSCAN mask/LTX 时 Vivado 会报告没有受支持的 debug core；这不是 bitstream 缺失。debug 板测必须同时加载匹配 LTX 并设置用户扫描链 0100。
- 完整设计 CDC 报告仍含基线 `core0/rst0`、`rst1` 和 MIG calibration reset 的 Critical；采集实例 Critical CDC=0。锁步核/基线复位不在本轮范围，不以采集门禁名义掩盖。

## 4. 结论

采集控制面、FT601 帧完整性、4096 点连续窗口、六路现有协议事件、FIFO 无丢失、绝对时间轴、失败结束判定和恢复均在实板成立。e416 最终候选的复烧、ILA 和 FT601 STATUS 未发现相对完整功能板测的新问题。当前阻止正常 pass 的唯一采集闭环缺口是缺少真实 `program_done`；物理 UART 和全局基线 CDC 作为独立问题记录。
