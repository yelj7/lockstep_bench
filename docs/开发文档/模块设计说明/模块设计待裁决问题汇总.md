# 1. 已裁决跨模块规则

## 1.1 任务级归属规则

裁决：任务产物、日志、证据和报告均直接归属任务目录，不引入中间执行层级。

实现口径：

- 所有模块以 `task_id` 作为任务产物、日志、证据、报告的核心归属键。
- 目录统一使用 `tasks/<task_id>/logs/`、`tasks/<task_id>/evidence/`、`tasks/<task_id>/waveform/`、`tasks/<task_id>/reports/`。
- UI 不提供相关入口、字段或摘要。
- 文档、接口、JSON 和日志格式不再包含相关字段。

## 1.2 `pass` 强制证据集合

裁决：只要跑通程序即可认为 `pass`。

强制条件：

1. 烧写成功记录。
2. 回读校验通过记录。
3. 运行控制真实返回或快照。

非阻塞记录：

- VCD 波形。
- 协议解析结果。
- 采集文件。
- 故障注入记录。
- warning。

这些记录缺失时必须在 `report.json` 中体现状态，但不阻断 `pass`。

## 1.3 测试模式故障注入

裁决：故障注入非必须。

实现口径：

- 测试模式仍可显示故障注入入口。
- 未配置或跳过故障注入时记录为 `skipped`。
- 故障注入不作为运行、报告或 `pass` 的强制前置。

## 1.4 回读失败后的运行控制

裁决：回读失败不允许运行。

实现口径：

- 回读失败、回读不一致或缺少回读通过记录时，M04/M07 必须关闭运行 gate。
- UI 不提供强制运行入口。
- M13 不得在回读失败时生成 `pass`。

# 2. 资源和镜像规则

## 2.1 `resources/manifest.json` schema

裁决：v1 使用 M03 文档中固化的 manifest schema。

关键字段：

- `schema_version`
- `resource_pack_id`
- `resource_pack_version`
- `defaults`
- `board_profiles`
- `debug_adapters`
- `report_templates`
- `protocol_rules`

所有资源路径均相对 `resources/`；任务和报告只保存资源逻辑 ID、版本和摘要。

## 2.2 板卡 profile schema

裁决：参考 `C:\Users\zzz\Desktop\profile.json`。

纳入字段：

- `profileName`
- `host`
- `gdbPort`
- `tclPort`
- `jtagKhz`
- `interfaceConfigPath`
- `targetConfigPath`
- `targetDebugToolPath`
- `ramBaseAddress`
- `defaultRunAddress`
- `resetStrategy`
- `runAfterDownload`

profile 内部可保存固化资源路径；任务、报告、UI 不保存或显示这些物理路径。

## 2.3 镜像格式

裁决：支持 ELF、SREC，以及 02 软件设计说明书提到的三类镜像；支持自动检测，不在范围内就报错。

实现口径：

- 支持 ELF。
- 支持 SREC 及 `.srec`、`.s19`、`.s28`、`.s37`、`.mot`。
- 支持原始二进制 `.bin`。
- 覆盖 02 设计书中的：带段信息的可执行镜像、原始二进制镜像、带地址记录的文本镜像。
- 自动检测优先基于文件内容，后缀辅助确认。

## 2.4 协议规则资源

裁决：先作为占位。

实现口径：

- `protocol_rules` 可登记为 `placeholder`。
- 协议解析缺失或占位不阻断 `pass`。
- 报告中记录协议解析状态。

# 3. 错误、日志和报告规则

## 3.1 错误编码

裁决：无统一格式，用合适的编码即可。

实现口径：

- 代码实现可采用字符串、分段码或枚举。
- 同一版本内错误编码含义必须稳定。
- 日志和报告保留错误编码、来源模块、阶段、建议和上下文摘要。

## 3.2 未修复错误

裁决：有未修复的错误就不能忽略。

实现口径：

- 未修复阻断错误禁止依赖流程继续。
- 未修复阻断错误禁止生成 `pass`。
- UI 不提供“忽略未修复阻断错误并继续通过”的动作。

## 3.3 warning

裁决：允许 warning。

实现口径：

- warning 写入日志。
- warning 写入 `report.json`。
- warning 不阻断 `pass`。
- 不设计 warning 对应的独立通过结论，结论仍为 `pass`。

## 3.4 报告格式

裁决：v1 先只要 report。

实现口径：

- v1 强制交付 `tasks/<task_id>/reports/report.json`。
- PDF/HTML/DOCX 等后续导出不作为 v1 验收条件。

# 4. 当前剩余开放项

用户已关闭会影响模块边界、核心接口、流程 gate、报告结论和资源 schema 的主要裁决问题。

后续若进入实现阶段，可在不改变上述规则的前提下自行决定：

- 具体错误编码命名。
- 日志文件轮转策略。
- 多段镜像记录是否拆成多个明细文件。
- profile 超时和重试默认值。
- report.json 的字段展示顺序。
