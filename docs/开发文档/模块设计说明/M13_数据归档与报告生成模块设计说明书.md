# 1. 设计定位

M13 是任务级事实归档和报告生成模块，负责把当前任务产生的真实产物归档到所属子工作区，并基于证据索引执行报告输入检查、`report.json` 生成和结论计算。

报告结论采用用户裁决口径：只要程序跑通所需强制证据成立即可 `pass`。VCD 波形、协议解析结果、采集文件和故障注入记录只作为记录项，不作为 `pass` 阻塞条件。

# 2. 设计输入与约束

输入：

- 当前任务。
- M02 任务目录和产物索引。
- M03 报告模板/schema 和资源摘要。
- M07 烧写、回读和运行控制记录。
- 后续采集、VCD、协议解析、故障注入等可选记录。

约束：

- v1 只要求生成 `report.json`。
- 有未修复的阻断错误时不能忽略后给出 `pass`。
- warning 允许通过，但必须进入报告记录。
- 报告不得记录导入前外部路径、profile 物理路径、模板物理路径或临时后端路径。

# 3. 模块职责与不负责事项

职责：

- 维护当前任务 `evidence/artifacts.json`。
- 汇总报告输入摘要。
- 校验 pass 强制证据是否成立。
- 生成 `report.json`。
- 在缺少强制证据、存在未修复阻断错误或流程失败时给出非通过结论。
- 将可选证据的存在、缺失、占位、跳过状态写入报告记录。

不负责：

| 事项 | 归属 |
|---|---|
| 任务输入管理 | M02 |
| 资源解析 | M03 |
| 流程 gate | M04 |
| 烧写回读运行 | M07 |
| VCD/协议解析产物生成 | M09-M12 |
| 错误解码 | M14 |

# 4. 数据对象与状态模型

| 对象 | 说明 |
|---|---|
| `ArchiveContext` | workspace/mode/task/operator |
| `ArtifactRecord` | 任务内产物路径、类型、摘要、状态 |
| `ReportInputSummary` | 报告输入摘要 |
| `ReportConclusion` | 结论、依据和阻断原因 |

报告结论：

| 状态 | 含义 |
|---|---|
| `pass` | 强制证据齐全且无未修复阻断错误 |
| `fail` | 强制步骤执行失败，例如回读不一致或运行失败 |
| `incomplete` | 强制证据缺失或流程未完成 |
| `blocked` | 资源、路径、未修复错误等阻断报告可信生成 |

不设置 warning 对应的独立通过结论；warning 记录在 `warnings` 数组中，结论仍可为 `pass`。

# 5. 对外接口与调用关系

| 接口 | 调用方 | 输出 |
|---|---|---|
| `attachArtifact(task_id, artifact)` | 各业务模块 | 更新后的产物索引 |
| `buildReportInput(task_id)` | UI/M04 | 报告输入摘要 |
| `validatePassEvidence(task_id)` | M04/UI | pass 强制证据检查结果 |
| `generateReport(task_id)` | UI/M04 | `report.json` 路径和结论 |
| `getReportSummary(task_id)` | UI | 报告摘要 |

# 6. 主流程与异常流程

归档流程：

1. 业务模块提交产物记录。
2. M13 校验产物路径位于当前任务目录内。
3. 计算摘要、大小、生成时间。
4. 写入 `evidence/artifacts.json`。

报告生成流程：

1. 读取任务、资源摘要和产物索引。
2. 读取烧写成功记录、回读校验通过记录、运行控制真实返回或快照。
3. 检查未修复阻断错误。
4. 汇总可选记录：VCD、协议解析、采集、故障注入、warning。
5. 计算结论。
6. 写入 `reports/report.json`。

异常流程：

| 异常 | 结论/处理 |
|---|---|
| 烧写成功记录缺失 | `incomplete` |
| 回读校验通过记录缺失 | `incomplete` |
| 回读不一致 | `fail` |
| 运行真实返回或快照缺失 | `incomplete` |
| 未修复阻断错误 | `blocked` 或 `fail`，按错误来源确定 |
| VCD 缺失 | 记录为 `not_available`，不阻断 pass |
| 协议解析缺失 | 记录为 `not_available`，不阻断 pass |
| 故障注入未配置 | 记录为 `skipped`，不阻断 pass |

# 7. 文件、目录、数据库或配置影响

报告文件：

```text
tasks/<task_id>/
  evidence/
    artifacts.json
  reports/
    report.json
```

`report.json` 建议结构：

```json
{
  "report_id": "rpt_20260706210000_0001",
  "task_id": "t_research_20260706205900_0001",
  "mode": "research",
  "conclusion": "pass",
  "required_evidence": {
    "program_write": "passed",
    "readback_verify": "passed",
    "run_control": "passed"
  },
  "optional_records": {
    "vcd_waveform": "not_available",
    "protocol_analysis": "not_available",
    "fault_injection": "skipped"
  },
  "warnings": [],
  "resource_snapshot": {}
}
```

# 8. 错误处理、日志与证据规则

pass 强制条件：

1. 烧写成功记录。
2. 回读校验通过记录。
3. 运行控制真实返回或快照。

以下内容不作为 pass 阻塞条件：

- VCD 波形。
- 协议解析结果。
- 采集文件。
- 故障注入记录。
- warning。

以下内容仍会阻断 pass：

- 强制证据缺失。
- 回读失败或不一致。
- 运行控制失败或无真实返回/快照。
- 当前任务存在未修复阻断错误。
- 报告写入失败或模板/schema 无法使用。

# 9. 与 UI 的数据合同

UI 可显示：

- 报告结论。
- 三类强制证据检查结果。
- 可选记录状态。
- warning 列表。
- 未修复阻断错误列表。
- `report.json` 路径。

UI 不要求 PDF/HTML/DOCX 导出入口作为 v1 验收项。

# 10. 测试场景与验收标准

| 场景 | 验收标准 |
|---|---|
| 三类强制证据齐全 | 结论可为 `pass` |
| warning 存在 | 结论仍可为 `pass`，warning 入报告 |
| VCD 缺失 | 结论仍可为 `pass` |
| 协议解析缺失 | 结论仍可为 `pass` |
| 回读失败 | 结论不得为 `pass` |
| 运行快照缺失 | 结论不得为 `pass` |
| 未修复阻断错误 | 结论不得为 `pass` |
| v1 报告生成 | 只验收 `report.json` |

# 11. 设计假设

1. `report.json` 是 v1 唯一强制交付物。
2. PDF/HTML/DOCX 等导出能力后续扩展，不影响 v1 pass 口径。
3. 可选记录缺失必须在报告中显式呈现，不能静默省略。

# 12. 待用户裁决问题

无。pass 强制证据、warning、可选记录和报告格式已按用户裁决固化。
