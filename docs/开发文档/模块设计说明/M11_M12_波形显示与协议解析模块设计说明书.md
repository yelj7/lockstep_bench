# 1. 设计定位

M11 是嵌入上位机的自研波形显示模块，负责读取当前任务固定位置的 VCD、显示统一时间轴、展示 9 组协议组、支持缩放平移和波形细节展开。M12 是嵌入上位机的自研协议解析模块，负责根据固化 trace 配置解析 VCD，生成事务级协议结果、关键行为列表、诊断摘要和诊断明细。

M11/M12 均作为 `software/lockstep_host` 下的原生 C++/Qt 子模块实现，不复用 `logic_analyzer` 或 Python 分析工具作为独立进程。DSView 可作为交互行为和解码思路参考，但实现、数据合同和 UI 结构按本系统需求自研。

职责分界：M10 负责采集帧处理并生成 VCD/schema；M12 负责协议解析并生成 analysis；M11 只负责显示并在 analysis 缺失或过期时触发后台解析；M13 负责登记产物和报告记录；M14 负责统一错误记录。

# 2. 设计输入与约束

固定任务产物：

```text
tasks/<task_id>/waveform/lockstep_trace.vcd
tasks/<task_id>/waveform/lockstep_trace_schema.json
tasks/<task_id>/waveform/lockstep_trace_analysis.json
```

核心约束：

| 约束 | 设计口径 |
|---|---|
| 每任务一份 VCD | 当前任务只保存一份 `lockstep_trace.vcd`，新采集覆盖旧 VCD 和对应 schema/analysis |
| VCD 信号形态 | 标准 VCD 只包含一条采样主信号 `lockstep_trace_sample[511:0]` |
| 时间轴权威 | 以 VCD 时间戳为权威，采样序号只用于诊断和对齐提示 |
| 自动读取 | M11 只索引当前任务固定文件名，不提供手动导入 VCD 的用户接口 |
| 固化配置 | 通道映射、协议解析、显示配置和板卡 trace 连接配置均固化在 `resources/lockstep_trace/` |
| 缺字段处理 | 未采集协议或缺少字段时必须按实际问题显示 `missing`、`not_captured` 或 `partial`，不得生成伪事务 |
| 解析输出 | M12 对外文件合同仅为 `waveform/lockstep_trace_analysis.json` |
| 报告结论 | 波形和协议解析是记录项，不作为 v1 `pass` 强制证据 |

固定显示 9 组协议组：

| 顺序 | 组 ID | 默认显示 |
|---:|---|---|
| 1 | `ahb` | AHB 事务级结果 |
| 2 | `uart` | UART 事务级结果 |
| 3 | `spi` | SPI 事务级结果 |
| 4 | `can` | CAN 事务级结果 |
| 5 | `i2c` | I2C 事务级结果 |
| 6 | `eth` | ETH 事务级结果 |
| 7 | `usb` | USB 事务级结果 |
| 8 | `jtag` | JTAG 事务级结果 |
| 9 | `mismatch` | 5-bit mismatch 解析结果 |

# 3. 模块职责与不负责事项

M11 职责：

- 自动检测当前任务 `waveform/lockstep_trace.vcd` 是否存在。
- 读取 `lockstep_trace_schema.json` 和 `lockstep_trace_analysis.json`。
- 在 analysis 缺失、过期或与 VCD/schema 摘要不一致时，后台请求 M12 重新解析。
- 显示 9 组事务级结果，支持下拉展开字段级波形。
- 支持统一时间轴下的水平缩放、平移、定位和关键行为跳转。
- 在缺失、部分解析或字段不可用时显示明确状态。

M12 职责：

- 校验 VCD 是否符合 `lockstep_trace_sample[511:0]` 合同。
- 读取固化 trace profile、通道映射、协议字段定义和显示配置。
- 基于 VCD 时间戳和字段变化计算协议事务、关键行为和异常。
- 生成 `lockstep_trace_analysis.json`。
- 聚合高频诊断问题，主动提交结构化错误事件给 M14。
- 将 M14 返回的 `error_id` 写回 analysis 的诊断摘要。

不负责事项：

| 不负责事项 | 归属模块 |
|---|---|
| 采集硬件控制和触发配置 | M09 |
| 采集帧接收、校验和 VCD/schema 生成 | M10 |
| 任务目录创建和路径挂接 | M02 |
| 固化资源校验和资源摘要 | M03 |
| 流程 gate 和阶段状态 | M04 |
| 产物索引和报告生成 | M13 |
| 错误记录存储和用户提示映射 | M14 |

# 4. 数据对象与文件合同

## 4.1 VCD 合同

标准 VCD 必须提供一条 512-bit 主信号：

```text
lockstep_trace_sample[511:0]
```

M11/M12 不根据 VCD 内部信号名推断协议字段。字段拆分只来自 `lockstep_trace_schema.json` 和固化资源中的通道映射。若 VCD 缺少该主信号、宽度不是 512 bit、时间戳无法解析或存在位宽冲突，M12 必须输出 error 级诊断并上报 M14。

## 4.2 schema 合同

`lockstep_trace_schema.json` 是本次采集的产物说明文件，不是用户可编辑配置。它记录本次 VCD 的采集合同、资源快照、VCD 摘要、字段映射版本和采集来源。

推荐字段：

```json
{
  "schema_version": "1.0",
  "task_id": "t_research_20260708100000_0001",
  "vcd_file": "waveform/lockstep_trace.vcd",
  "vcd_sha256": "<sha256>",
  "sample_signal": "lockstep_trace_sample",
  "sample_width": 512,
  "timescale": "1 ns",
  "trace_profile_id": "trace.noelv.lockstep_512",
  "trace_profile_version": "1.0.0",
  "channel_map_sha256": "<sha256>",
  "capture_started_at": "2026-07-08T10:00:00+08:00",
  "capture_finished_at": "2026-07-08T10:00:05+08:00"
}
```

## 4.3 analysis 合同

`lockstep_trace_analysis.json` 是 M12 的唯一对外解析产物。M11、M13 和报告 UI 均读取该文件，不依赖 M12 的跨模块内存模型。

推荐字段：

```json
{
  "schema_version": "1.0",
  "task_id": "t_research_20260708100000_0001",
  "input": {
    "vcd_file": "waveform/lockstep_trace.vcd",
    "schema_file": "waveform/lockstep_trace_schema.json",
    "vcd_sha256": "<sha256>",
    "schema_sha256": "<sha256>"
  },
  "status": "complete",
  "time_base": {
    "unit": "ns",
    "start_time": 0,
    "end_time": 100000
  },
  "groups": [],
  "key_behaviors": [],
  "diagnostic_summary": [],
  "diagnostic_details": []
}
```

`groups[]` 保存 9 组协议的事务级结果、字段可用性和可展开字段列表；`key_behaviors[]` 保存按时间排序的关键总线、外设行为或 mismatch 发生；`diagnostic_summary[]` 保存可上报给 M14 的聚合问题并带 `error_id` 引用；`diagnostic_details[]` 保存字段级和协议级详细原因，供 UI/report 排查使用。

# 5. 固化资源结构

源码树中资源建议放置在：

```text
software/lockstep_host/resources/lockstep_trace/
  noelv_lockstep_512/
    trace_profile.json
    board_connection.json
    channel_map.json
    protocol_decoders.json
    display_profile.json
```

打包后归属 `install_root/resources/lockstep_trace/`，由 M03 的 manifest 登记逻辑 ID、版本、相对路径和摘要。

资源分工：

| 文件 | 内容 |
|---|---|
| `trace_profile.json` | 512-bit trace 合同、默认 9 组、资源版本和兼容性 |
| `board_connection.json` | 采集硬件与板卡 trace 连接说明 |
| `channel_map.json` | `lockstep_trace_sample[511:0]` 到协议字段的位段映射 |
| `protocol_decoders.json` | 协议字段依赖、完整性门槛、行为类型、显示枚举 |
| `display_profile.json` | 默认行顺序、合并字段、值显示格式、展开规则 |

资源 JSON 描述结构、字段和显示策略；C++ 代码实现实际 decoder 逻辑。后续如果硬件映射更新，应更新固化资源和资源摘要，不在任务中保存手工导入的映射文件。

# 6. 关键硬件映射

采集主信号固定为 512 bit。当前已知 mismatch 位段来自 NOEL-V comparator 的 `equals[4:0]`，并被采集硬件放入 `lockstep_trace_sample[506:502]`。

固定 mismatch 映射：

| sample bit | mismatch bit | 语义 |
|---:|---:|---|
| 506 | `mismatch[4]` | `ahb_master_output_mismatch` |
| 505 | `mismatch[3]` | `irq_output_mismatch` |
| 504 | `mismatch[2]` | `debug_output_mismatch` |
| 503 | `mismatch[1]` | `trace_output_mismatch` |
| 502 | `mismatch[0]` | `counter_output_mismatch` |

多 bit mismatch 解析规则：

- 任一 mismatch bit 从 0 变 1 时创建或并入当前 mismatch 事件。
- 事件在所有已激活 mismatch bit 均回到 0 时结束。
- 活动窗口内新增 mismatch bit 合并到同一个 `mismatch_event`。
- `mismatch_event.items[]` 必须完整列出所有 bit、语义、起止时间和持续时间。
- UI 默认行必须完整显示多项 mismatch，可采用同一事件块内多行文本或可展开明细，不得只显示第一项。

# 7. UI 展示规则

默认视图只显示 9 组顶层事务行。每组左侧提供展开按钮，展开后显示字段级波形行。

显示规则：

- 协议组名称作为一级行，字段名称作为二级行，字段名视觉上向右缩进。
- 缩进只影响名称栏，不改变波形区同一时间轴。
- 可合并 bus 字段优先合并展示，例如 `AHB/data[127:0]`。
- bus 合并字段和地址字段默认使用带 `0x` 前缀的十六进制。
- 枚举和状态字段显示语义名。
- 单 bit 信号显示电平波形。
- 未采集协议显示 `not_captured`；缺少必要字段显示 `missing_fields` 或 `partial_decode`。
- 关键行为列表按 `start_time` 顺序展示，并与主时间轴联动定位。

时间轴和性能规则：

- 水平方向支持自由缩放和平移。
- 缩放后波形、事务块、关键行为和 mismatch 事件必须仍按 VCD 时间戳正确对齐。
- v1 即按 viewport 渲染设计，内部使用变化段、时间索引和可视区抽样，避免全量逐点绘制。
- v1 不把持久索引缓存作为对外合同；需要时可在 M11 内部构建可丢弃的内存索引。

# 8. 协议解析范围

M12 以标准协议 decoder 框架设计，按采集硬件实际提供的字段进行解码。未采集或缺字段时输出明确诊断，不生成伪事务。v1 先闭合核心行为，后续可以在不改变 analysis 合同的前提下扩展更完整的协议行为。

v1 核心行为：

| 协议 | 核心行为 |
|---|---|
| AHB | read/write transaction、burst、stall、error response |
| UART | TX/RX byte、frame error |
| SPI | transfer、MOSI/MISO word、CS frame |
| CAN | frame、id、data、ack、error mark |
| I2C | start/stop、address、read/write、ack/nack、data |
| ETH | frame start/end、ethertype、payload byte summary、rx/tx error |
| USB | packet、PID、endpoint、data、reset |
| JTAG | TAP state、IR/DR scan、TDI/TDO stream |
| mismatch | 5-bit 同时解析、复合 mismatch 事件 |

字段完整性门槛由 `protocol_decoders.json` 描述。例如 AHB 缺少地址、方向或响应字段时不能生成完整 transaction；可以保留字段波形和诊断明细，并将协议组状态标为 `partial_decode`。

# 9. 错误处理与 M14 协作

M12 是波形/协议问题的错误源，负责聚合诊断并主动上报 M14。M14 不遍历 `lockstep_trace_analysis.json` 来发现 M12 问题，只接收 M12 提交的 `ErrorEvent` 并生成 `ErrorRecord`。

诊断分层：

| 层级 | 保存位置 | 用途 |
|---|---|---|
| `diagnostic_summary[]` | analysis | UI/report 摘要，带 M14 `error_id` 引用 |
| `diagnostic_details[]` | analysis | 字段级、协议级调试原因 |
| `ErrorRecord` | M14 | 统一错误存储、顶部状态、阻断判断和日志 |

严重级别建议：

| 情况 | 级别 |
|---|---|
| 无 VCD、协议组未采集、字段缺失、decoder 能力未覆盖、无事务但波形可显示 | `warning` |
| VCD 不可读、VCD 解析失败、timescale/var 失败、512-bit 合同冲突、资源规则自相矛盾 | `error` |
| 任务目录、waveform 目录或日志目录不可写 | `fatal` 或阻断错误 |

高频问题必须聚合后上报。例如同一协议的同一字段缺失只上报一条 summary，不得每个采样点生成一条 M14 错误记录。M14 宜提供批量写入接口，M12 在一次解析结束后批量提交诊断摘要。

# 10. 主流程

采集后流程：

1. M10 生成或覆盖 `lockstep_trace.vcd`。
2. M10 生成或覆盖 `lockstep_trace_schema.json`。
3. M13 登记 VCD 和 schema 产物。
4. M12 读取 VCD/schema 和固化资源，生成 `lockstep_trace_analysis.json`。
5. M12 将诊断摘要批量上报 M14，并把 `error_id` 写入 analysis。
6. M13 登记 analysis 产物。
7. M11 打开波形界面时读取固定三文件并显示。

打开波形界面流程：

1. M11 通过 M02 获取当前任务目录。
2. 检查 `waveform/lockstep_trace.vcd` 是否存在。
3. 若 VCD 不存在，显示无波形状态并记录 warning。
4. 若 VCD 存在但 analysis 缺失、过期或摘要不匹配，后台调用 M12 解析。
5. M12 完成后 M11 重新加载 `lockstep_trace_analysis.json`。
6. UI 默认展示 9 组事务级结果，用户按需展开字段波形。

# 11. 测试场景与验收标准

| 场景 | 验收标准 |
|---|---|
| 当前任务无 VCD | M11 显示无波形状态，M12/M14 记录 warning，不阻断 pass |
| VCD 主信号缺失 | M12 输出 error 诊断并上报 M14 |
| VCD 主信号不是 512 bit | M12 输出合同冲突 error |
| analysis 缺失 | M11 后台触发 M12，完成后刷新显示 |
| analysis 与 VCD 摘要不一致 | M11 触发重新解析 |
| 默认打开波形页 | 显示 9 组顶层事务行 |
| 展开 AHB | bus 字段合并展示，data/address 默认十六进制 |
| 协议未采集 | 对应组显示 `not_captured` 和明确原因 |
| 协议缺必要字段 | 对应组显示 `partial_decode` 和缺字段列表 |
| 多 bit mismatch | 同一 mismatch 事件完整列出所有 mismatch 项 |
| 时间轴缩放 | 波形边沿、事务块和关键行为保持时间对齐 |
| 高频缺字段 | M12 聚合为少量 summary 错误，不刷爆 M14 |

# 12. 设计假设与待裁决问题

设计假设：

1. M10 按本设计统一生成一条 512-bit packed VCD 信号。
2. M03 能校验并提供 `resources/lockstep_trace/` 下的固化 trace profile。
3. M14 提供单条和批量错误记录接口；若批量接口尚未实现，M12 内部仍需先聚合再提交。
4. 后续协议解码扩展通过固化资源和 C++ decoder 版本演进完成，不引入任务级手工导入配置。

待裁决问题：无。VCD 形态、任务产物、固化配置、默认显示、错误上报和 decoder 范围已按用户裁决固化。
