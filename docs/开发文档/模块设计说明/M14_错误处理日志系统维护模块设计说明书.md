# 1. 设计定位

M14 负责统一接收各业务模块产生的结构化错误事件，完成错误解码、用户提示映射、处理建议、日志格式化、错误记录写入、顶部阻断原因更新和系统维护检查。

各业务模块在最接近错误源的位置做校验并产生错误事件；M14 不替代业务校验，不猜测失败原因，不吞错继续执行。

# 2. 设计输入与约束

- 错误编码无统一格式强制要求，采用适合实现的稳定编码即可。
- 有未修复的阻断错误就不能忽略后形成成功状态。
- warning 允许通过，但必须记录。
- v1 报告只要求 `report.json`。
- 任务日志写入当前任务目录；系统诊断日志写入子工作区 `system_logs/`。

# 3. 模块职责与不负责事项

职责：

- 定义统一错误事件结构。
- 维护错误编码表或等价映射。
- 映射用户提示、开发定位和建议。
- 写入任务日志和系统日志。
- 为 UI 提供最近错误、阻断原因和 warning 摘要。
- 执行维护检查：资源、模板、协议占位、路径、数据库/JSON、证据摘要、日志目录。

不负责：

- 不修复业务错误。
- 不把错误降级为成功。
- 不删除证据、报告或历史日志。
- 不计算报告结论。

# 4. 数据对象与状态模型

统一错误事件：

```json
{
  "error_id": "err_20260706210000_0001",
  "code": "RESOURCE_MANIFEST_MISSING",
  "severity": "error",
  "source_module": "M03_RESOURCE_CONFIG",
  "failure_stage": "validate_manifest",
  "task_id": "t_research_20260706205900_0001",
  "request_id": "req_xxx",
  "message": "资源清单缺失",
  "suggestion": "检查安装包 resources/manifest.json",
  "blocking_flag": true,
  "status": "open"
}
```

严重级别：

| 级别 | 含义 |
|---|---|
| `info` | 普通记录 |
| `warning` | 可继续但需记录 |
| `error` | 当前步骤失败 |
| `fatal` | 系统或工作区不可继续 |

错误状态：`open`、`resolved`、`superseded`。不设计“忽略后继续”作为消除阻断错误的状态。

# 5. 对外接口与调用关系

| 接口 | 调用方 | 输出 |
|---|---|---|
| `recordError(event)` | 各模块 | 错误记录 |
| `writeLog(entry)` | 各模块 | 日志写入结果 |
| `getBlockingErrors(task_id)` | M04/M13/UI | 未修复阻断错误 |
| `getWarnings(task_id)` | M13/UI | warning 列表 |
| `runMaintenanceCheck(scope)` | UI/启动流程 | 维护检查结果 |

# 6. 主流程与异常流程

错误事件主流程：

1. 业务模块提交错误事件。
2. M14 校验必填字段。
3. 解析错误编码或使用兜底分类。
4. 生成用户提示和处理建议。
5. 写入任务日志或系统日志。
6. 若为阻断错误，更新当前任务最近阻断原因。
7. 返回错误记录 ID。

维护检查流程：

1. 按范围检查资源、路径、数据库/JSON、模板、协议规则、日志目录和证据摘要。
2. 对失败项生成错误事件。
3. 输出维护检查摘要。

异常流程：

| 异常 | 处理 |
|---|---|
| 缺少错误编码 | 使用实现层兜底分类 |
| 未注册错误编码 | 保留原编码和上下文 |
| 日志目录不可写 | 返回 fatal 或系统诊断错误 |
| 原始上下文过大 | 摘要入日志，完整内容归档 |

# 7. 文件、目录、数据库或配置影响

日志文件：

```text
tasks/<task_id>/logs/operation.log
tasks/<task_id>/logs/debug_access.log
tasks/<task_id>/logs/error.log
system_logs/system.log
```

日志格式不强制统一到某一编码样式，但单次实现内必须字段稳定，至少包含：

- 时间戳。
- 级别。
- 来源模块。
- task_id，可为空。
- 操作或失败阶段。
- 消息。
- error_id 或 request_id。

# 8. 错误处理、日志与证据规则

阻断错误未修复时：

- M04 不允许继续依赖阶段。
- M07 不允许在回读失败后运行。
- M13 不允许生成 `pass`。

warning：

- 写入日志。
- 进入 `report.json` 的 `warnings`。
- 不阻断 `pass`。

必须阻断通过结论的情况：

- 烧写成功记录缺失或烧写失败。
- 回读校验通过记录缺失、回读失败或不一致。
- 运行控制真实返回或快照缺失。
- 当前任务存在未修复阻断错误。
- 报告写入失败。

不阻断通过结论但需记录的情况：

- VCD 波形缺失。
- 协议解析结果缺失。
- 采集文件缺失。
- 故障注入未配置或跳过。
- warning。

# 9. 与 UI 的数据合同

UI 可读取：

- 最近错误/阻断原因。
- 未修复阻断错误列表。
- warning 列表。
- 维护检查摘要。
- 日志摘要。

UI 不提供“忽略未修复阻断错误并继续通过”的动作。

# 10. 测试场景与验收标准

| 场景 | 验收标准 |
|---|---|
| 标准错误事件 | 写错误记录和操作日志 |
| 未注册编码 | 使用兜底提示但保留上下文 |
| 阻断错误 | 更新最近错误并阻断依赖流程 |
| warning | 写日志、入报告、不阻断 pass |
| 回读失败错误 | M07 运行 gate 关闭 |
| 未修复阻断错误 | M13 不生成 pass |

# 11. 设计假设

1. 错误编码表由 M14 或实现层统一维护。
2. `system_logs/` 不作为任务报告 pass 强制证据。
3. 任务日志可被报告引用为记录。
4. 数据库和 JSON 两种实现路径保持相同错误语义。

# 12. 待用户裁决问题

无。错误编码、未修复错误、warning 和报告交付格式已按用户裁决固化。
