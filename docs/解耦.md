# 程序运行与采样生命周期解耦

## 零、执行优先级与精简边界

当前 UI 是产品功能和接口的唯一真源。实施前必须建立 `UI action -> controller -> domain service -> transport/RTL command -> evidence -> test` 可达性矩阵，并据此精简 Host 底层与采样硬件。

保留条件只有两类：当前 UI 明确提供的功能；支撑这些功能正确运行、终止收口、错误恢复、状态查询、传输和证据校验所必需的内部机制。其他没有 UI 可达路径的 public API、领域状态、配置字段、CLI、证据字段、协议命令、RTL 端口/参数、源码和未来预留全部删除。仅有单元测试调用不构成保留理由。

内部原语与产品功能必须分开：`haltreq` 可作为“程序终止”的私有实现，`STOP_CAPTURE` 可作为终止/错误恢复/无触发收口的内部命令，ARM/GET_STATUS/START_EVENT_STREAM/FIFO drain 可作为自动采集内部协议；它们不得形成 Halt、独立停止采集等用户功能。产品、领域、证据和状态统一使用 Terminate/Terminated，不使用 Halt/Halted。

程序侧只保留“程序运行”“程序复位并运行”“程序终止”三种用户功能，不新增暂停、继续或独立停止采集功能。
底层可以继续使用 RISC-V Debug Module 的 haltreq 实现“程序终止”，但该机制不得在界面、状态、证据和领域模型中表现为用户可操作的 Halt 功能。程序终止后，下一次运行必须执行“程序复位并运行”。
一、基本原则
程序生命周期与采样生命周期完全独立：
program_result:
  running
  program_done
  terminated
  run_timeout
  reset_error
  debug_error
  fatal_error

capture_result:
  armed
  triggered
  window_complete
  no_trigger_program_done
  terminated_before_trigger
  termination_truncated
  trigger_timeout
  watchdog
  overflow
  drop
  sequence_gap
  upload_timeout
  fatal_error
采样完成不代表程序完成，程序完成或终止也不自动代表采样成功。

上位机的正常产品判定要求程序结果为 program_done、采样结果为 window_complete、sample_count=4096、身份/CRC/序号/字节数/状态链全部有效且无 overflow/drop/sequence gap。任何单侧成功都不能替代产品判定。上位机不读取 benchmark golden，也不输出 benchmark PASS；benchmark 脚本仅在功能开发阶段独立比较仿真或实板产物与 golden。合法的跨窗口 partial transaction 不自动表示采集失败，其是否满足特定测试预期由外部 benchmark 脚本判断。

结束原因必须使用唯一 v4 合同值：WINDOW_COMPLETE、HOST_TERMINATE、TRIGGER_TIMEOUT、WATCHDOG、OVERFLOW、HARD_TIMEOUT、NO_TRIGGER_PROGRAM_DONE、FATAL_ERROR。Host 将 HOST_TERMINATE 按是否已有真实 trigger 分为 terminated_before_trigger/termination_truncated；upload_timeout 是 Host 上传阶段本地结果，不伪造硬件结束原因，禁止沿用 v3 的 PROGRAM_DONE=0。
二、采样模块上电状态
FPGA 上电或采样模块复位后，自动进入 CONFIGURED_READY：
装载安全的默认采样配置。
可以接受配置。
可以接受 ARM。
不上传上一轮残留数据。
不自动触发。
不要求用户点击“开始采集”或“保存采样配置”。
下发采样配置时，上位机自动把当前配置保存到任务中。
每次程序运行都必须生成新的 capture_id，旧 capture 的迟到帧不得进入新任务。
三、运行前 ARM 顺序
必须保留并强化当前“ARM 后才启动程序”的顺序。
普通“程序运行”：
确认上一轮采样已经完全收口
→ 下发或确认采样配置
→ ARM_CAPTURE(capture_id)
→ 等待 ARM_ACK
→ 确认采样域、事件存储和上传通路 ready
→ 发送程序运行命令
“程序复位并运行”：
执行程序 reset
→ 确认 reset 完成
→ 等待采样模块处于 CONFIGURED_READY
→ 下发或确认采样配置
→ ARM_CAPTURE(capture_id)
→ 等待 ARM_ACK 和 capture_ready
→ 发送程序运行命令
如果程序复位会同时复位采样模块，必须先完成程序复位，再 ARM。ARM 或 ready 确认失败时禁止启动程序，避免错过位于程序入口附近的 PC 触发点。
PC 触发必须使用有效执行或退休 PC，不能因为程序终止状态下 PC 静止在入口地址而误触发。
四、正常采样结束
第一次有效触发发生在绝对采样索引 T：
window_start = T - 2047
window_end   = T + 2048
采样模块必须在写入 T+2048 后停止接受新样本和新变化事件，随后只排空、上传已经保存的数据。
program_done 不得作为已触发采样的结束条件，也不得作为 FPGA 采样 RTL 的直接输入。程序可以在采样完成后继续运行，采样不得继续等待程序结束。
五、程序终止处理
上位机的“程序终止”必须同时编排程序控制和采样收口，但不能直接杀死采样线程，也不能在收到 CAPTURE_END 前复位采样模块。
1. 触发前终止
发送程序终止命令
→ 确认程序已终止
→ 向采样模块发送内部收口命令
→ 生成 terminated_before_trigger
→ sample_count=0
→ 接收并校验 CAPTURE_END
→ 返回 CONFIGURED_READY
不得伪造触发点、VCD 或采样通过结果。
2. 触发后、采满前终止
真实 trigger_index 不得改变。该情况是后触发窗口被截短，不是“重新解释为触发点靠后”。

有效触发与已经同步到采样域的程序终止请求同拍时，触发优先：先接受并保存触发点 T 和该拍所有变化，再执行终止冻结。此时 actual_window_end=T、actual_sample_count=2048，结果为 termination_truncated；只有终止在 T 前一拍已经生效时才是 terminated_before_trigger。
必须记录：
requested_window_start = T - 2047
requested_window_end   = T + 2048
actual_window_end      < requested_window_end
actual_sample_count    < 4096
window_complete        = false
termination_truncated  = true
处理顺序：
发送程序终止命令
→ 确认程序已终止
→ 采样模块冻结当前实际有效窗口
→ 停止接受新样本和事件
→ 排空并上传已有数据
→ 接收并校验 CAPTURE_END
→ 持久化部分 VCD、sidecar 和终止状态
→ 返回 CONFIGURED_READY
部分采样可以作为诊断数据保存，但不得作为完整 benchmark 采样通过。
3. 采样完成后终止
采样结果已经冻结，程序终止不得修改采样数据：
发送程序终止命令
→ 确认程序已终止
→ 等待本轮采样上传完成
→ 校验 CAPTURE_END 和统计
→ 返回 CONFIGURED_READY
六、程序结束与未完成外设行为
程序打印 program_done 不代表硬件外设已经静止。以下行为可能继续：
UART 移位寄存器仍在发送。
SPI、I2C、CAN 事务尚未结束。
AHB 访问仍有未完成响应。
DMA、定时器或其他自主 PL 外设仍在运行。
处理规则：
program_done 到达且采样已经触发：继续采到固定 window_end，不得提前复位。
program_done 到达但尚未触发：程序结果为 program_done，采样结果收口为 no_trigger_program_done，sample_count=0；不得等待不可达的 PC，不得伪造 trigger、VCD 或采样通过结果。
用户主动终止且采样正在进行：允许生成部分窗口。
可靠 `protocol_busy` 只用于判断 partial/incomplete 和终止收口，不得让正常已触发采样提前冻结；正常采样始终写到固定 `window_end`。只有用户程序终止路径允许在 `window_end` 前冻结为 `termination_truncated`。
不得为了等待外设而超过原始 window_end。
外设事务跨越实际窗口边界时，协议解析必须标记为 partial，不得伪造完整事务。
没有可靠 busy 信号时，不得仅凭短时间无边沿声称外设已经静默。

PROGRAM_RUN_DONE 由 Host 从所选 MI_02 实时串口确认。下发本轮 run 前必须清空接收缓冲，从 run 成功下发后开始逐行解析；只接受去除 CR/LF 后按大小写不敏感比较完全等于 PROGRAM_RUN_DONE 的行。禁止任意子串、旧 marker 或 Dhrystone 启发式判定。合并 VCD 中恢复出的同名 UART 行只能交叉验证采样与解析，不能替代实时程序结果。
七、上传与复位顺序
所有完成、失败和终止路径统一执行：
进入 FROZEN，禁止新数据写入
→ 立即锁存实际窗口和结束原因
→ 进入 DRAINING，等待由最大 CDC/边沿检测/事件合并延迟推导的固定内部 quiet_guard_cycles
→ 确认同步器与合并管线无 pending 项
→ 排空连续样本、sparse 事件和所有数据帧
→ 发送并排空 CAPTURE_END
→ 接收 CAPTURE_END
→ 校验 capture_id、sample_count、字节数和序号
→ 校验 overflow、drop、sequence gap
→ 保存 raw capture、sidecar、VCD 和状态证据
→ 清除本轮缓存
→ 返回 CONFIGURED_READY
不得在上传完成前复位采样模块，也不得使用 QThread::terminate() 中断正常采集收口。
quiet guard 不是通过无边沿时间猜测外设静默；guard/drain 无法收口时必须记录 fatal_error/overflow，禁止提前发送 CAPTURE_END。CAPTURE_END 后不得再发送本轮数据帧，Host 完整校验前不得开始下一轮 ARM。
内部 STOP_CAPTURE 可以作为程序终止、错误恢复和无触发收口命令，但不得暴露为用户可见的“停止采集”按钮。
八、需要修复的现有缺口
根据当前代码，必须修复：
stopProgram() 当前只终止程序，没有通知正在运行的采样会话。
程序终止不会向采样模块发送 STOP_CAPTURE。
采样线程只在应用析构时收到 interruption。
当前 STOP_CAPTURE 仅用于错误恢复。
当前 Host 把 HostTerminate 统一判为采样失败，尚不能表达有效的部分采样。
当前内部仍使用 ProgramOperation::Halt、RunState::Halted 和 haltTarget()；领域语义应统一为 Terminate/Terminated，底层 haltreq 只作为实现细节。
程序终止成功后必须继续保持下一次运行使用“复位并运行”的现有语义。
九、必须新增的测试
ARM_ACK 和 capture_ready 前程序绝不启动。
普通运行与复位并运行均不会错过入口 PC 触发。
触发前程序终止。
触发后第一个 sample、窗口中间和 window_end 前一拍终止。
采样完成后程序终止。
程序终止与触发同拍。
program_done 在触发前、触发同拍、采样中和采样完成后到达。
不可达 PC、分支绕过 PC 和程序死循环。
终止后的部分窗口保存真实 trigger index 和实际 sample count。
上传期间终止不会丢数据或提前复位。
外设 busy、跨窗口 UART/SPI/I2C 事务和 partial 标记。
连续两轮 ARM 的 capture ID、缓存和迟到帧隔离。
任意 overflow、drop、sequence gap 或上传不完整均不得生成产品成功结论或开发门禁通过。
十、验收标准
程序启动前采样模块已经 ARM 并明确 ready。
程序运行、程序终止和采样窗口分别拥有独立、可审计的结果。
正常采样严格在 window_end 停止，不等待 program_done。
程序终止能正确收口触发前、采样中和采样后三种状态。
部分采样不伪造完整窗口或移动触发点。
所有数据上传、校验和持久化完成后，采样模块才返回 CONFIGURED_READY。
下一次运行可以自动配置、ARM 并触发，不受上一轮残留状态影响。

十一、UI 功能真源与精简验收

必须输出并逐项审查可达性矩阵。每个保留的底层模块、公共接口、状态、配置字段、证据字段和 RTL 命令都必须能追溯到 UI 功能或必要内部机制；每个删除项必须同时清理调用、构建清单、资源、测试和文档。

明确删除或收紧：Halt/Halted 产品语义、暂停/继续、独立开始采集、独立保存采样配置、独立停止采集、无真实探针协议源、未被当前 UI 使用的配置预留和产品 CLI。下发采样配置时自动保存到 task；程序运行和程序复位并运行自动 ARM 采样。

精简完成后不得出现“UI 无入口但底层仍公开可调用”的产品能力。必要的内部命令必须是私有或受限接口，只能由规定的编排路径调用，并有自动测试证明不会被当作独立用户功能。
